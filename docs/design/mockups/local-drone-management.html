Local drone management and eviction recovery

Exact terminal outcomes for the operator removing a seat, the removed session receiving its terminal signal, and later worktree cleanup. Eviction ends the current Borg seat; it does not delete project files or silently enroll a replacement.

Operator evicts a drone

Success 
MCP tool call  borg_evict-drone
{
  "label": "builder-7a12",
  "cube_id": "326ea162-1234-4abc-8def-1234567890ab"
}

Tool result
Removed builder-7a12 from cube borg-mcp.
The seat credential is revoked. The session will stop after its next Borg request.
The worktree and project files were not deleted. Activity history remains attributed to the removed seat.
After its work is merged, run `borg cleanup` to review whether the worktree can be pruned.
Required outcome
  • Name the human-readable drone label and cube.
  • Distinguish seat removal from file deletion.
  • Do not claim attribution was anonymized unless the server actually does so.

Local display warning: The server committed this change, but Borg could not refresh this worktree's saved seat metadata. Local role details may be stale. Do not retry the reassignment.

Run `borg_regen` to refresh the server-authoritative role. If this session still shows the previous role, restart this agent session once. Do not re-assimilate or repeat the management request.
Required outcome

  • Say that local display details may be stale, never that the server result is uncertain.
  • Prohibit retry, then give one bounded recovery sequence: regen first, session restart only if still stale.

Removed session wakes

Terminal 
[CUBE-EVICTED] This seat was removed from cube borg-mcp.

Borg has stopped listening for activity for this seat. Do not retry this request or restart the loop.

Your worktree and project files are unchanged. Finish any local file safety checks, then end this agent session.

To rejoin later, start a new session and use a new invitation from the server operator. Do not re-assimilate from this evicted session.
Required outcome
  • Lead with the terminal state, not the HTTP code.
  • Give one unambiguous recovery path.
  • Keep protocol detail available to logs, not as the primary instruction.

Cleanup reviews the worktree

Safe dry-run 
operator $ borg cleanup

borg cleanup report
  PRUNABLE  ~/.borg/worktrees/borg-mcp/builder-7a12
            seat removed; worktree clean; branch merged into origin/main

1 prunable, 0 kept.
Dry-run: nothing was deleted. Re-run with `--prune` to remove this worktree.
Required outcome
  • Translate the server verdict into user-relevant language.
  • Keep dry-run and destructive action visibly distinct.
  • Never make a bare 410 status the primary explanation.

Behavior rules

Reassignment and eviction are authoritative after the pinned local authority returns their typed success verdicts; no second network read is required to prove commit. Generic authentication failures, expired credentials, network errors, trust mismatches, and unknown responses remain non-destructive. Drone management is authorized by a server-derived principal with cube management permission. Coordinator and Queen are workflow labels, not authorization capabilities.

Implementation acceptance

  • src/index.ts matches the eviction and reassignment results above and makes no anonymized-attribution claim.
  • src/tool-manifest.ts describes retained attribution and cube management permission; it removes the hosted-era Owner-scoped promise and does not present a role label as an authorization boundary.
  • src/remote-client.ts describes the local cube-scoped management contract and retained attribution without hosted RLS, ownership, or anonymization claims.
  • src/drone-lifecycle.ts gives the removed session one recovery path: stop this session, then rejoin later from a new session with a new invitation.
  • A confirmed mutation is never reported as failed because ancillary local metadata refresh failed. The result keeps the success copy, adds the exact non-retry warning above, and performs no post-commit network request.
  • Self-role metadata refresh updates only the exact original seat identity under lock (worktree, credential reference, cube, and drone). A concurrent replacement is an honest no-op, never an overwrite.
