Local drone management and eviction recovery

Exact terminal outcomes for the operator removing a seat, the removed session receiving its terminal signal, and later worktree cleanup. Eviction ends the current Borg seat; it does not delete project files or silently enroll a replacement.

Operator evicts a drone

Success 
operator $ borg_evict-drone label=builder-7a12 cube_id=326ea162

Removed builder-7a12 from cube borg-mcp.
The seat credential is revoked. The session will stop after its next Borg request.
The worktree and project files were not deleted. Activity history remains attributed to the removed seat.
After its work is merged, run `borg cleanup` to review whether the worktree can be pruned.
Required outcome
  • Name the human-readable drone label and cube.
  • Distinguish seat removal from file deletion.
  • Do not claim attribution was anonymized unless the server actually does so.

Removed session wakes

Terminal 
[CUBE-EVICTED] This seat was removed from cube borg-mcp.

Borg has stopped listening for activity for this seat. Do not retry this request or restart the loop.

Your worktree and project files are unchanged. Finish any local file safety checks, then end this agent session.

To rejoin later, start a new session and use a new invitation from the server operator. Do not re-assimilate from this evicted session.
Required outcome
  • Lead with the terminal state, not the HTTP code.
  • Give one unambiguous recovery path.
  • Keep protocol detail available to logs, not as the primary instruction.

Cleanup reviews the worktree

Safe dry-run 
operator $ borg cleanup

borg cleanup report
  PRUNABLE  ~/.borg/worktrees/borg-mcp/builder-7a12
            seat removed; worktree clean; branch merged into origin/main

1 prunable, 0 kept.
Dry-run: nothing was deleted. Re-run with `--prune` to remove this worktree.
Required outcome
  • Translate the server verdict into user-relevant language.
  • Keep dry-run and destructive action visibly distinct.
  • Never make a bare 410 status the primary explanation.

Behavior rules

Eviction is authoritative only after the pinned local authority returns the typed eviction verdict. Generic authentication failures, expired credentials, network errors, trust mismatches, and unknown responses remain non-destructive. Reassignment should use the same label-and-cube-first language and should confirm the destination role by name rather than UUID alone.
