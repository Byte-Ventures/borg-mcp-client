Interaction mockup / borg-mcp-client

Local management authority

Management actions use the selected local client's server grant. Coordinator and Queen remain workflow roles; changing a drone's role does not change server authorization.

Managing client records a decision

Success 
MCP tool call  borg_decide
{
  "topic": "repository-topology",
  "decision": "Active development occurs in the public client, server, and shared repositories.",
  "rationale": "Keep the local product boundary explicit."
}

Tool result
Recorded ratified decision on "repository-topology" in cube "borg-mcp".
Cite it via borg_decisions; it surfaces in borg_regen.

MCP tool call  borg_decisions
{ "topic": "repository-topology" }

Tool result
repository-topology: Active development occurs in the public client, server, and shared repositories.
Required outcome
  • The selected parent client credential has cube management access.
  • Success is proven by immediate readback, not by the write response alone.
  • No drone-session credential is treated as a management credential.

Known cube, insufficient access

Typed denial 
MCP tool call  borg_decide
{
  "topic": "repository-topology",
  "decision": "..."
}

Tool result
[LOCAL-MANAGE-REQUIRED] This session cannot record a decision in cube "borg-mcp" because the selected local client does not have cube management access.

Coordinator and Queen are workflow roles; they do not grant server permissions. Nothing was recorded. Do not retry this request from this session.

Use a session whose local client already has management access to this cube.
Required outcome
  • Shown only after the server recognizes the cube and an authenticated read/write client lacking manage access.
  • The marker stops blind agent retries; the zero-mutation sentence removes ratification ambiguity.
  • Recovery uses an already-managing session. It does not cite a nonexistent grant command.

Another management action is denied

Operation-specific 
MCP tool call  borg_evict-drone
{
  "label": "builder-7a12",
  "cube_id": "326ea162-1234-4abc-8def-1234567890ab"
}

Tool result
[LOCAL-MANAGE-REQUIRED] This session cannot remove "builder-7a12" from cube "borg-mcp" because the selected local client does not have cube management access.

Coordinator and Queen are workflow roles; they do not grant server permissions. No drone was removed. Do not retry this request from this session.

Use a session whose local client already has management access to this cube.
Required outcome
  • Reuse the authority explanation and recovery path.
  • Name the attempted operation and target.
  • State the exact outcome: "No drone was removed."

Unknown or unauthorized cube

Non-disclosure 
MCP tool call  borg_decide
{ "topic": "repository-topology", "decision": "..." }

Tool result
Error: Borg server request failed (HTTP 404)
Required outcome
  • Do not emit [LOCAL-MANAGE-REQUIRED] for an unknown, foreign, or no-grant cube.
  • Do not reveal whether the cube, client, topic, or target exists.
  • Never reinterpret a generic 403 or 404 from another route as a management denial.

Implementation acceptance

  • Only server-declared management operations use the selected parent client credential. Unsupported local tools remain explicitly unsupported.
  • The server returns typed 403 ACCESS_DENIED only for an authenticated, known-cube read/write client that lacks manage access. Unknown, no-grant, and foreign-cube requests remain 404.
  • The client maps only a management operation's typed denial to [LOCAL-MANAGE-REQUIRED]. It does not map generic 403/404 responses.
  • The denied action performs zero mutation, is not retried, and never falls back to Cloud.
  • Managing success plus readback and same-cube non-manage denial plus no-write are end-to-end release gates.
